# RFC-017 — Enforced Judge and Conflict Results

## Objective

Ensure the canonical reasoning state always contains an inspectable Judge
decision and a meaningful Conflict result. This corrects an incomplete
implementation of the existing Judge and conflict-detection requirements; it
does not add a new reasoning stage or alter the public session lifecycle.

## Judge Contract

For every completed reasoning round, Judge output must contain:

- `validated_claims` — claims supported by the supplied artifacts;
- `conflicts` and `unresolved_conflicts` — explicit lists, empty only when
  none are found;
- `conflict_summary` — an explanation of whether material conflicts exist and
  why;
- `synthesis` — a concise comparative decision based on the recorded evidence
  and perspectives.

The Judge must not return a generic placeholder. If a provider returns valid
JSON without this contract, the application performs one structured repair
request. A second incomplete result fails the session with an explicit
provider error rather than persisting a misleading empty Judge or Conflict
node.

## Conflict Node

Conflict remains a derived graph node after Judge, preserving the documented
pipeline and session lifecycle. Its data is copied from the Judge's
model-generated `conflict_summary` and conflict lists; orchestration must not
invent a generic status string.

## Compatibility

- Existing fields (`validated_claims`, `conflicts`, `unresolved_conflicts`,
  `synthesis`) remain available.
- `conflict_summary` is additive and is included in the canonical JSON state.
- Markdown and graph views consume the same canonical Judge result.
