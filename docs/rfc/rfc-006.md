# RFC-006 — Production Readiness

## Objective

Prepare the MVP for deployment.

## Requirements

Environment variables only.

No hardcoded secrets.

Graceful API failures.

Health endpoints.

- GET /health/live confirms that the process is running. It must not call the
  reasoning provider.
- GET /health/ready confirms that required local dependencies, including
  SQLite/session storage, are usable. It must not make a paid or external
  provider request.

Request timeouts.

- Provider HTTP requests have a bounded timeout.
- A reasoning session has a bounded total execution time.
- Frontend polling has a bounded request timeout and displays a safe,
  actionable failure when the session expires or fails.

Rate limiting.

- Limit session creation and session-start requests by client IP.
- Define a concrete limit and a clear 429 response.
- Configure trusted proxy handling so deployment behind a reverse proxy does
  not collapse all clients into one IP or trust arbitrary forwarded headers.

Structured logging.

- Log session_id, stage, provider, duration, and outcome.
- Never log API keys, authorization headers, full user question/context, or
  raw provider responses.

Production build succeeds.

README deployment section updated.

Docker build succeeds.

.env.example updated.

## Provider and Evidence Policy

OpenAI native Responses API is the recommended production provider because
Evidence retrieval uses hosted web search and retains only URLs returned as
search citations.

AI/ML API may be used for non-grounded reasoning stages, but until equivalent
hosted-search support is verified it must not return plausible-looking,
unverified evidence URLs. The UI/API must state that grounded evidence is
unavailable rather than fabricating a citation.

When AI/ML API is configured, the Evidence stage uses the separately configured
`AIMLAPI_EVIDENCE_MODEL` search-capable model with the same `AIMLAPI_API_KEY`.
Only URLs returned in that response's `citations` or `search_results` may be
shown as grounded evidence. If that request fails, the fallback may return
model-derived engineering considerations, each with a null URL and explicit
`unverified` quality; it must never be presented as web-grounded evidence or
increase evidence quality.

## Ephemeral Deployment Recovery

Render free storage is ephemeral, so the MVP must support importing a completed
Thesision JSON export. Import validates the expected session shape, assigns a
new local session ID, and restores the graph, evidence, confidence, exports,
and replay state without re-running the reasoning provider. Imported files are
not treated as prompts or executable configuration.

A restored completed session may also create an automatic continuation child.
The child preserves prior reasoning, appends up to three new rounds, and
creates a new conclusion without asking the user to repeat the question. The
parent session remains immutable.

## Docker Requirements

- Run the application as a non-root user.
- Do not copy .env or secrets into the image.
- Make the listening port configurable.
- Provide a container healthcheck using the liveness endpoint.
- Store SQLite/session data in a documented persistent volume path.
